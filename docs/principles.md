# 原理文档

## 处理原理

字幕任务分为三个阶段：

1. 转录
   - 使用本地 `openai-whisper` CLI 直接处理视频
   - 输出原始 SRT
2. 优化
   - 将原始 SRT 解析为字幕条目
   - 分块后交给 LLM 做语法、标点和可读性优化
   - 保留时间轴不变，只替换文本内容
3. 翻译
   - 对优化后的字幕再次分块
   - 逐块翻译成多个目标语言
   - 每种目标语言输出一个独立 SRT 文件

## 为什么要分块

LLM 对输入长度有限制，长字幕不能一次性送入模型，因此需要分块：

- `chunkSize`
  - 控制每块最多包含多少条字幕
- `overlap`
  - 控制相邻块共享多少条上下文

这样做有两个目的：

- 避免 prompt 过长
- 在分块边界处保留上下文，降低前后句断裂风险

## 为什么要做合规替换

某些词汇可能触发模型拒答，或者在翻译后需要统一替换为特定表达。为此在调用 LLM 前会：

1. 把命中的敏感词替换为占位符
2. 将占位符连同恢复映射一起记录
3. LLM 返回后再恢复成目标文本

这样可以降低两类风险：

- 直接触发模型安全拒答
- 术语翻译不一致

## 为什么会有回退策略

优化和翻译并不要求每个块都完全成功。当前实现采用“局部失败优先回退”的原则：

- LLM 调用失败
- 返回块数不匹配
- 输出高度相似，疑似没有翻译
- 输出泄露 prompt 或占位符

遇到这些情况时，会优先回退到原始或 sanitize 后文本，而不是中断整条流水线。

原因是：

- 用户通常更需要“有结果但局部不完美”
- 大批量任务时，单块失败不应拖垮整批任务

## 为什么主输出是 `.srt`

流水线最终会把优化后的 LLM 字幕复制为 `<basename>.srt`，而不是默认把某个翻译结果覆盖成主文件。

这样做的原因是：

- 原语言优化字幕通常最适合直接回看或再加工
- 多语言翻译结果已经各自独立输出，不需要抢占主文件名
- 常见播放器对同名 `.srt` 的自动匹配更友好

## 状态流转原理

任务状态的典型流转如下：

- `queued`
- `transcribing`
- `optimizing`
- `translating`
- `completed`

异常情况下：

- 用户暂停或主动中断：`paused`
- 执行失败：`failed`

这些状态全部由 `TaskStore` 持久化，视图层只负责展示，不自己推导状态。
